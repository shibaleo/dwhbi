# 4PIC model

4PIC（4 Practice-Information Cycle）は、データ活用を構造化するモデルである。

- 情報(information)とは意味を付与されたデータ(文字列・数値など)である。
- データ活用を、log・analyze・hope・adjustの4つの実践(4 practices)で定義する。
- 各実践から生まれるデータに、実績・推定・願望・目標という4種類の意味を付与し、4つの情報(4 information)として定義する。

## 4practices（4つの実践）

| # | 実践 | 具体的な行動 | 創出されるデータの意味 |
|---|------|-----------|-------------|
| 1 | log | 日々のログ取得、データ入力 | 実績 |
| 2 | analyze | 実績を集計し、分析する | 推定 |
| 3 | hope | 目的達成に必要な数値を設定 | 願望 |
| 4 | adjust | 現実的に達成可能な目標を立案 | 目標 |

- 仮にlogだけしか実行できなくても、実績データを残すことで、最低限の振り返り(analyze)が可能になる。
- 目標や計画がなければ機能しないようなモデルではなく、「まずは記録。そのあとは？」に応えるモデルである。

## 4information（4種類の情報）

4 practicesから生まれる4種類の情報。

### なぜ4種類なのか：2×2マトリクスによる分類

4 informationは、2つの軸の組み合わせから導出される。

| | 確実 | 不確実 |
|---|---|---|
| **客観** | actual | estimate |
| **主観** | target | wish |

- **客観/主観**: データが人間の意思を含むか否か
  - 客観: 事実または事実に基づく推定（意思を含まない）
  - 主観: 人間の意図・願望・決定を含む
- **確実/不確実**: データの確からしさ
  - 確実: 記録された事実、またはコミットされた目標
  - 不確実: 推定値、または調整前の願望

注意：targetは目標なのだから変わりうるのでは？

回答：当然変わりうるが、目標の変更はサイクルの更新を意味する。一度目標を決めた後は、その目標を前提として行動し、記録するという過程を説明するために 主観/確実 と定義している。

| # | 情報 | 説明 | 例 |
|---|-----------|------|-----|
| 1 | **actual** | 実際に記録された値 | 作業時間の記録 |
| 2 | **estimate** | 過去の実績から導出された推定値 | 推定作業時間 |
| 3 | **wish** | 主観的な目標。願望を含むものとして定義する。 | 週の作業時間目標、月間目標 |
| 4 | **target** | 現時点で手に入るすべてのデータをもとに、wishを実現可能な値に調整した目標 | 調整後の作業時間、現実的なスケジュール |

### practiceとinformationの対応関係

情報を軸に整理すると、4 practicesは情報を変換する処理であり、4 informationはその入出力である。

```
actual ─(analyze)─→ estimate ─(hope)─→ wish ─(adjust)─→ target ─(log)─→ actual ...
```

| 入力 | 実践 | 出力 |
|------|----------|------|
| (target) | log | actual |
| actual | analyze | estimate |
| estimate, 外部要因 | hope | wish |
| wish | adjust | target |

- サイクルとして循環し、targetに基づいて行動・記録(log)を行い、新たなactualが生まれる。
- 実践と情報の対応は概念的な大枠であり、1:1に限定されない。1つのanalyzeから複数のestimateが生まれることもあれば、複数のwishを1つのtargetに統合することもある。

### actualとestimateの関係

- **actual（実績）** は過去に記録された事実である。
- **estimate（推定）** は、actualを集計・分析して導出した推定値である。

| 情報 | 時間軸 | 性質 |
|------|--------|------|
| **actual** | 過去 | 事実（変更不可） |
| **estimate** | 未来 | 推定（不確実性あり） |

- analyzeは単一の処理ではなく、パターン発見・異常検知・傾向把握などを含む。これらの分析結果をもとにestimateを算出する。
- 単純な平均だけでなく、傾向・季節性・外れ値、クロスドメインの影響などを考慮することで、estimateの精度は向上する。
- estimateはwish・targetと異なり、人間の意思を含まない客観的な推定値である。この客観性が、wish→targetの調整における判断基準となる。
- analyzeで得られたインサイト（傾向、異常など）は、hope段階でユーザーがwishを検討する際の参考情報となる。

### wishとtargetの違い

- 一般的に目標は感覚・勘・願望に基づいて設定されることが多い。
- 本モデルではそれを考慮し、主観的に設定された目標を **wish（願望）** として定義する。
- **target（目標）** は、wishを過去の実績とそれに基づく推定(estimate)に基づいて調整した、実現可能な値である。

| 情報 | 性質 | データ処理 |
|------|------|-----------|
| **wish** | 主観・願望が主 | estimate・外部要因（達成したい目的）・主観的判断をもとに人間が決める |
| **target** | 実現可能性が制約 | 必須（estimateなど、複数の要因を考慮） |

```
estimate: 過去3ヶ月の平均は80時間（実績ベースの推定）
wish: 推定は80時間だが、来月は100時間働きたい（estimateを参照した上での願望）
target: wishと推定を踏まえ、現実的に達成可能な85時間を設定（調整後の目標）
```

adjustの実践では、願望だけでなく実績を踏まえた「実現可能な目標」を立案する。

### なぜwishとtargetを分離するのか

組織においても個人においても、「計画を第一に」という姿勢で計画を立てながら、結果として計画倒れや炎上を招くケースは普遍的に見られる。これは多くの場合、願望と実現可能な目標が混同されていることに起因する。

本モデルでは、この普遍的な失敗パターンを構造に組み込む：

- **wish**: 達成したい目的や主観的に「現実的」と考える値。estimateを参照しても、願望が含まれる限りwishである
- **target**: estimateに基づく客観的な制約を反映した、実現可能な値

wishをtargetに変換するadjustの実践を明示することで、「計画が現実的かどうか」を検証するプロセスをモデル化している。

### targetを算出する意義

targetは「調整済みの実現可能な目標」として算出されるが、現実には：

- 想定外の出来事により達成できない
- 運や外部要因により余裕で達成できる

といったケースが発生する。これは失敗ではなく、**targetとactualの乖離を分析する材料**となる。

```
target: 85時間
actual: 70時間（未達）→ なぜ15時間の乖離が生じたか？
actual: 95時間（超過）→ なぜ10時間多く達成できたか？
```

この乖離分析により：
- estimateの算出ロジックを改善できる
- adjustの調整基準を精緻化できる
- 外部要因の影響度を把握できる

targetを明示的に記録することで、サイクルを回すたびにモデル全体の精度が向上する。

## モデルの価値

### 紙とペンでも実践可能

- このモデルは現代のIT技術を前提としたものではない。
- このモデルにおける情報と実践の構造はコンピュータ出現以前から存在していたと考えられる。

紙とペンだけでも、以下のように4practicesを実践できる：

1. **log**: 紙に行動実績（いつ何をしたか）を記録する → actual が生まれる
2. **analyze**: 過去の記録をもとに、特定作業の平均所要時間を手計算で算出する → estimate が生まれる
3. **hope**: 次の行動目標を立てる。必ずしも紙に書くとは限らないが、達成したい数値は頭の中にある → wish が生まれる
4. **adjust**: 現実的な制約を考慮して調整する。例えば、一日20時間作業するような計画は一般的な生活と両立しない → target が生まれる

しかし3.hopeと4.adjustを明確に区別して行うことは難しく
- 計画倒れ
- 過大な目標設定

となることが多い。

### なぜ体系化が必要か

| 課題 | モデルによる解決 |
|------|------------------------|
| 記録だけで終わる | 記録だけでも構わないが、そのあとの実践を明示する |
| データの関係性が不明 | 4informationとして情報を分類・整理 |
| 何を見ればいいかわからない | actual を中心に分析 |
| 改善につながらない | 主観的な目標を願望(wish)として定義し、実現可能な目標(target)を提案 |

### 4informationの組み合わせパターン

4PICは4つの情報すべてを揃えることを強制しない。actualを必須とすると、残り3つ（estimate, wish, target）の有無により8パターンが存在する。

| パターン | 保持する情報 | 説明 |
|----------|-------------|------|
| 記録のみ | actual | 最小構成。記録を残すだけでも、後から振り返る材料になる |
| 振り返り | actual + estimate | 実績から推定を導出し、傾向を把握する |
| 願望駆動 | actual + wish | 願望と実績を比較。adjustを経ていないため乖離が生じやすい |
| 目標駆動 | actual + target | 外部から与えられた目標に対して実績を記録・比較する |
| 願望設定 | actual + estimate + wish | 推定を参照して願望を設定するが、調整はしないので目標達成の意識が薄れる |
| 合理的目標 | actual + estimate + target | 推定から直接目標を設定。効率的だが、願望がないと動機づけが弱まる可能性がある |
| 経験則依存 | actual + wish + target | 願望を調整して目標化するが、推定なしで行う。経験則に依存 |
| 完全サイクル | actual + estimate + wish + target | 4practicesすべてが機能し、継続的な改善が可能 |

- **actualは必須**: どのパターンでも、記録（actual）がなければ始まらない
- **順序は問わない**: wishから始めてもよいし、外部からtargetが与えられることもある
- **段階的に拡張可能**: 記録だけから始めて、必要に応じて他の情報を追加できる

## 関連する概念・フレームワーク

4PIC modelは独自の概念ではなく、既存のフレームワークや概念と共通点がある。

| 概念 | 共通点 |
|------|--------|
| **PDCA** | サイクル構造（Plan-Do-Check-Act）。4PICはデータの種類（4information）を明示的に分類する点が異なる |
| **OKR** | 目標の階層化（Objectives and Key Results）。wishとtargetの分離は、野心的な目標と測定可能な成果指標の分離と類似 |
| **GTD** | タスク・行動管理（Getting Things Done）。logによる記録とanalyzeによる振り返りの構造が共通 |
| **Quantified Self** | データの記録・分析による自己理解。actualの蓄積とanalyzeという基本思想が共通 |

これらの概念に馴染みがあれば、4PIC modelの理解の助けとなる。GTDやQuantified Selfは個人向けに設計されたフレームワークだが、4PICとの共通点は主体を問わない。
