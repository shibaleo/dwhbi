# ADR-001: リリース戦略

## ステータス

採用

## コンテキスト

LIFETRACERは8つの外部サービスと連携する必要がある。各サービスは異なる認証方式を持ち、複雑さが異なる。効率的に開発を進めるための戦略が必要。

## 決定

段階的にサービスを追加する。API Token系のシンプルなサービスから始め、OAuth系の複雑なサービスへと進む。

### バージョニング

セマンティックバージョニングを採用：

```
MAJOR.MINOR.PATCH

MAJOR: 破壊的変更（スキーマ変更等）
MINOR: サービス追加
PATCH: バグ修正、ドキュメント更新
```

### リリース計画

#### Phase 0: API Token系（シンプル）

| バージョン | サービス | 認証方式 |
|-----------|---------|---------|
| v0.1.0 | Toggl Track | API Token |
| v0.2.0 | Trello | API Key + Token |
| v0.3.0 | Airtable | PAT |

#### Phase 1: OAuth系

| バージョン | サービス | 認証方式 |
|-----------|---------|---------|
| v0.4.0 | Fitbit | OAuth 2.0 |
| v0.5.0 | Tanita Health Planet | OAuth 2.0 |
| v0.6.0 | Google Calendar | OAuth 2.0 |
| v0.7.0 | TickTick | OAuth 2.0 |
| v0.8.0 | Zaim | OAuth 1.0a |

#### v1.0.0: 正式リリース

- 全8サービス対応完了
- 管理ダッシュボード安定
- ドキュメント完備

## サービス追加チェックリスト

### 必須

- [ ] rawテーブルスキーマ定義
- [ ] マイグレーションSQL
- [ ] 認証処理実装
- [ ] API取得実装
- [ ] DB書き込み実装
- [ ] 同期スクリプト実装
- [ ] 管理UI設定フォーム
- [ ] Vault保存/読み取り
- [ ] GitHub Actions統合
- [ ] ドキュメント更新

### 推奨

- [ ] ユニットテスト
- [ ] サービス固有README
- [ ] トークンリフレッシュ対応（OAuth系）

## 結果

- 認証複雑度の低い順に実装することで、学習曲線を緩やかにできる
- 各マイナーバージョンで1サービス追加という明確な目標
- v1.0.0 で全サービス対応完了の明確なゴール
