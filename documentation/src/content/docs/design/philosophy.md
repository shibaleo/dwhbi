# 設計哲学

## 基本思想：変化を記録することで自分自身を理解する

LIFETRACERは、60年以上の運用を想定した包括的な個人データ統合・分析システムである。その根底にあるのは、**行動変容ではなく客観的自己分析**という目的意識だ。

### 人生の微分

数学における微分が、関数の瞬間的な変化率を捉えるように、LIFETRACERは人生という関数を「変化の記録」を通じて理解しようとする。

静的な状態（「私は何者か」）を直接問うのではなく、変化のパターン（「私はどのように変わっているか」）を観測することで、間接的に自己を理解する。これは物理学が現象を直接ではなく、測定可能な変化を通じて理解するアプローチと同様である。

### なぜ「理解」であり「変容」ではないのか

多くのライフログツールは行動変容を目的とする。しかしLIFETRACERは、変容の前提となる**理解**に焦点を当てる。

- 「なぜ」という答えのない問いから、「どのように」という観測可能な問いへ
- 主観的な内省から、客観的なデータ分析へ
- 意志による変化から、パターンの発見へ

## 核心概念：Designative Liability

### 定義

> **Designative Liability（指示責任）**：記号に対して特定の解釈を指示し、その指示に従って行動するルールを設定することで生じる、そのルールへの責任。指示者自身および指示を受ける者の双方を拘束する。

例えば、APIの仕様書に「`datetime: 20200104123456` はUTC 2020-01-04T12:34:56 を意味する」と記載し、実際にそのように動作させること。これがDesignative Liabilityである。

この責任を負う者は：
1. 指示の妥当性・一貫性に責任を持つ
2. 自らもその指示に拘束される
3. 指示を受ける者（利用者）をも拘束する

### Semantic Binding（意味による拘束）

Designative Liabilityと対をなす概念として、**Semantic Binding**がある。

> **Semantic Binding**：指示されたルールに従う義務。指示を受け入れた者が負う拘束。

両者の関係は包含的である：

```
┌─────────────────────────────────────────┐
│         Semantic Binding               │
│      （意味による拘束を受ける）           │
│                                         │
│    ┌───────────────────────────────┐   │
│    │  Designative Liability       │   │
│    │  （指示責任を負う）            │   │
│    │                               │   │
│    │  API設計者、DB設計者、         │   │
│    │  サービス提供者                │   │
│    └───────────────────────────────┘   │
│                                         │
│    API利用者、エンドユーザー             │
│                                         │
└─────────────────────────────────────────┘
```

| 立場 | Designative Liability | Semantic Binding |
|------|:---------------------:|:----------------:|
| 指示者（設計者） | ○ | ○ |
| 利用者 | × | ○ |

### 権限の非対称性

非対称なのは「拘束」ではなく「権限」である。

| 権限 | 指示者 | 利用者 |
|------|:------:|:------:|
| 指示を定義する自由 | ○ | × |
| 指示を変更する自由 | ○ | × |
| 拘束される義務 | ○ | ○ |

### 自由と義務の分離が認知負荷を下げる

| 立場 | 自由（定義する） | 義務（拘束される） | 認知負荷 |
|------|:---------------:|:-----------------:|:--------:|
| 指示者かつ利用者 | ○ | ○ | **高** |
| 利用者のみ | × | ○ | 低 |

**自由と義務の両方を同時に引き受けると認知負荷が高い。一方だけなら認知負荷は低い。**

これが、特化型サービス（Toggl、Fitbit等）の「使いやすさ」と、オールインワンツール（Notion等）の「設計疲れ」を説明する原理である。

## 設計原則：ログの分散、データの統合

### 従来のアプローチの問題

| アプローチ | 問題点 |
|-----------|--------|
| オールインワンツール | ユーザーが全領域でDesignative Liabilityを負う → 認知負荷が高い |
| 分散ツール（連携なし） | 各ツールがDesignative Liabilityを負うが、データが孤立 → 分析不可能 |

### LIFETRACERの解

> **ログの分散、データの統合**

```
┌─────────────────────────────────────────────────────────────────────┐
│                    特化型サービス（Designative Liability保持）        │
├─────────────┬─────────────┬─────────────┬─────────────┬─────────────┤
│   Toggl     │   Fitbit    │   Tanita    │    Zaim     │  Calendar   │
│  時間追跡   │  活動量     │   体組成    │   家計簿    │  予定管理   │
└──────┬──────┴──────┬──────┴──────┬──────┴──────┬──────┴──────┬──────┘
       │             │             │             │             │
       │      Semantic Binding（各サービスの定義に従う）        │
       │             │             │             │             │
       ▼             ▼             ▼             ▼             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                         LIFETRACER                                   │
│              統合レイヤー（Designative Liability引き受け）           │
│                                                                      │
│   ・異なるサービス間の関係性の定義                                   │
│   ・時間と健康の相関分析                                             │
│   ・支出パターンと活動の関係                                         │
│   ・長期的なトレンド分析                                             │
└─────────────────────────────────────────────────────────────────────┘
```

### 責任の配分

| レイヤー | Designative Liability | 内容 |
|----------|:---------------------:|------|
| 時間追跡 | Toggl | 「時間エントリー」「プロジェクト」の意味定義 |
| 健康データ | Fitbit, Tanita | 「心拍数」「体重」「体脂肪率」の意味定義 |
| 財務データ | Zaim | 「支出」「カテゴリ」の意味定義 |
| 予定管理 | Google Calendar | 「イベント」「予定」の意味定義 |
| **統合スキーマ** | **自分** | サービス間の関係性、相関分析の軸 |

これにより：
- 各領域では「拘束されるだけ」の低負荷状態を維持
- 統合レイヤーでのみ「指示者かつ利用者」の高負荷を引き受ける
- **負荷の局所化**と**分析の統合**を両立

## 思想的背景

### サルトルへの批判的応答

サルトルは「自由から逃れようとするのは悪しき信仰（mauvaise foi）だ」と言った。カフェのウェイターが役割に埋没するのは、自己欺瞞だと批判した。

しかしLIFETRACERの設計哲学は、これに対する批判的応答である：

> **Designative Liabilityの移譲は、悪しき信仰ではなく、比較優位に基づく合理的分業である。**

| 観点 | サルトル | LIFETRACER |
|------|----------|------------|
| 責任の移譲 | 悪しき信仰として批判 | 合理的分業として肯定 |
| 自由 | 引き受けるべき重荷 | 適切に配分すべきリソース |
| 帰結 | 自由を引き受けよ | 適切な専門家に移譲せよ |

サルトルは個人の実存に留まったが、LIFETRACERは「誰が負うべきか」という社会設計の問いに踏み込む。

> **自由の重荷は、適切な専門家に移譲されるべきであり、それは悪しき信仰ではない。**

### 新自由主義的「Responsibilization」批判との関係

社会学では、新自由主義が「自由」の名のもとに個人に責任を押し付ける現象を「Responsibilization（責任化）」と呼び、批判的に論じてきた。

しかし、これらの議論は**すべての責任移譲を批判的に見る**傾向がある。LIFETRACERの立場はより精緻である：

| 観点 | Responsibilization批判 | LIFETRACER |
|------|----------------------|------------|
| 責任移譲 | 批判的（押しつけ） | 中立的（配分の問題） |
| 評価基準 | 移譲自体が問題 | **誰が負うべきか**が問題 |
| 処方箋 | 責任を移譲するな | 適切な方向に移譲せよ |

「自由を与えるな」ではなく「**専門家が負うべき責任を素人に押し付けるな**」——これがLIFETRACERの立場である。

### オントロジー工学との接続

オントロジー工学では、データに意味を付与する構造を「オントロジー」と呼ぶ。

- **アッパーオントロジー**：普遍的概念（物、イベント、時間、場所）
- **ドメインオントロジー**：領域固有の概念（時間追跡、健康管理）
- **アプリケーションオントロジー**：特定システムの概念（Togglの「プロジェクト」）

特化型サービスは**アプリケーションオントロジーを事前に固定**している。これが「Ontological Commitment（存在論的コミットメント）」であり、Designative Liabilityをサービス提供者が引き受けていることの技術的表現である。

一方、Notionのような汎用ツールは**アッパーオントロジー的な抽象構造のみ**を提供し、ドメインオントロジーの構築をユーザーに委ねている。

## 実践的含意

### サービス選択の基準

サービスを選ぶとは、**Designative Liabilityをどこまで移譲するかの決定**である。

| 選択 | 意味 |
|------|------|
| 特化型サービスを選ぶ | Designative Liabilityを移譲し、Semantic Bindingのみを受け入れる |
| オールインワンツールを選ぶ | Designative Liabilityを自分で負い、将来の自分にSemantic Bindingを課す |

### データ設計の指針

1. **各領域のDesignative Liabilityは特化型サービスに委ねる**
   - 時間追跡はTogglの定義に従う
   - 健康データはFitbit/Tanitaの定義に従う
   - 自分で再定義しない

2. **統合レイヤーのDesignative Liabilityのみ自分で負う**
   - サービス間の関係性
   - 分析の軸と観点
   - 長期的なスキーマ設計

3. **rawデータとJSON保存の原則**
   - サービスのオントロジーが変わる可能性に備える
   - 構造化カラムとraw JSONの両方を保持
   - 将来の再解釈の余地を残す

### 60年運用への備え

LIFETRACERは60年以上の運用を想定している。この長期視点において、Designative Liabilityの概念は以下の意味を持つ：

- **サービスの消滅リスク**：特化型サービスが終了しても、rawデータがあれば再解釈可能
- **オントロジーの変化**：サービスの定義が変わっても、履歴として記録される
- **統合スキーマの進化**：自分が負うDesignative Liabilityは、時間とともに洗練される

---

## まとめ

LIFETRACERの設計哲学は、以下の原則に集約される：

1. **変化を記録することで自分自身を理解する**（人生の微分）
2. **Designative Liabilityは適切な専門家に移譲する**（比較優位の原則）
3. **ログの分散、データの統合**（負荷の局所化と分析の統合）
4. **自由と義務の分離が認知負荷を下げる**（認知的最適化）

これらは単なる技術的選択ではなく、**自己理解と責任配分に関する哲学的立場**である。
