# 非機能要件

ISO/IEC 25010（ソフトウェア品質モデル）に基づく品質特性を定義する。個人プロジェクトの特性上、要件を設けない項目も明記する。

## 1. 信頼性

| 項目 | 要件 |
|------|------|
| 稼働時間 | データベース・管理UIは常時稼働、同期処理はオンデマンド |
| 計画停止 | システム自体をユーザが所有するためなし |
| 障害復旧 | 手動復旧で可（自動復旧は不要） |
| 災害対策 | Supabaseの冗長性に依存、独自のDR対策は設けない |

## 2. 性能効率性

- 個人プロジェクトであり管理者=使用者のため、応答速度等の性能要件は設けない。
- ただし、GitHub Actionsの無料枠（月2,000分）を超過しないよう、同期スクリプトの実行時間は効率化を意識する。
- 単一ユーザー前提のため、スケールアウトは考慮しない。

## 3. セキュリティ

| 項目 | 要件 |
|------|------|
| 認証情報保護 | 保存時暗号化必須 |
| 通信 | HTTPS必須 |
| アクセス制御 | オーナーのみアクセス可能 |
| 監査ログ | 同期実行ログを保持 |

## 4. 使用性

- 管理者=使用者のため、アクセシビリティ対応は設けない。
- 多言語対応は設けない（日本語のみ）。

## 5. 保守性

| 項目 | 要件 |
|------|------|
| コード品質 | 型安全（TypeScript/Python型ヒント） |
| テスト | 主要機能のユニットテスト |
| ドキュメント | 設計ドキュメントの整備 |
| サービス追加 | 新規サービスの追加が容易な設計 |
| スキーマ変更 | マイグレーションによる安全な変更 |

## 6. 移植性

- 本プロジェクトは[021 設計哲学 - フォークを想定しない公開](/01-product/000-foundations/020-philosophy/021-design-philosophy#フォークを想定しない公開)ため、移植性の要件は設けない。

## 7. 互換性

| 項目 | 要件 |
|------|------|
| ブラウザ | Chrome/Firefox/Safari最新版 |
| API | 各サービスのAPI仕様変更に追従 |

## 8. 運用性

| 項目 | 要件 |
|------|------|
| デプロイ | Supabase / Vercel |
| 監視 | Supabaseダッシュボードで確認 |
| バックアップ | Supabaseの自動バックアップに依存 |
| ログ | 同期ログをDBに記録 |
| 運用工数 | 最小限（自動化優先） |

### インフラコスト方針

**金銭的コストゼロ**を原則とする。有料プランは使用しない。

- 各サービスの無料枠のみを使用（Supabase Free、Vercel Free、GitHub Actions無料枠）
- 容量不足時は別サービスの無料枠を併用（例：PostgreSQL互換のNeon）
- 有料プランへのアップグレードではなく、無料リソースの分散で対応

無料サービス依存に伴うリスクと代替サービスについては[430 外部サービス依存リスク](/01-product/400-operations/430-risk/service-dependency)を参照。
